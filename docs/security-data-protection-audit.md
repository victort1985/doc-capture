# Doc Capture — аудit защиты данных (DevSecOps)

Дата: 17.06.2026. Методология: реальные атаки/проверки на поднятом
сервере (PostgreSQL + NestJS) — SQL-инъекции, подмена JWT, brute-force,
проверка заголовков, прямой запрос к БД для проверки шифрования "at
rest" — плюс `npm audit` по фактическому дереву зависимостей с
разделением prod/dev. Не аудит по диагонали кода — там, где написано
"подтверждено", это воспроизведённое поведение, не предположение.

Severity: **Critical** — прямая утечка данных или полный обход защиты.
**High** — серьёзно повышает риск утечки при реалистичном сценарии.
**Medium** — заметный пробел в защите, не катастрофа сам по себе.
**Low** — рекомендация для усиления, не дефект.

---

## 1. Критические находки

### 1.1 Пароли от FTP/NAS хранятся и отдаются в открытом виде

Поле `password` в `storage_connections` — обычный `@Column()` без
шифрования. Подтверждено напрямую:

```
SELECT password FROM storage_connections WHERE name='ftp-secret-test';
 SuperSecretPlaintext1
```

И тем же открытым текстом оно уходит по API — `POST/GET
/api/storage/connections` возвращает `password` в ответе как есть.
Это значит: пароль от реального NAS/FTP бизнеса виден (а) всем, у кого
есть доступ к БД или её бэкапу, (б) в network-таб браузера у любого, кто
сидит под admin-сессией, (в) при любом перехвате трафика между админкой
и сервером — а трафик не шифрован вообще (см. 1.2).

Рекомендация: шифровать поле на уровне приложения (AES, ключ — отдельно
от БД, например деривация от `JWT_SECRET` или собственный
`ENCRYPTION_KEY` в `.env`), и не отдавать `password` обратно в GET/POST
ответах вообще (по аналогии с тем, как уже скрыт `passwordHash` у
пользователей через `select: false`) — админке для отображения списка
пароль не нужен, только индикатор "задан/не задан".

### 1.2 Весь трафик — обычный HTTP, TLS нигде не настроен

```
plain http STATUS=200
https attempt STATUS=000   # соединение не принято — TLS-порта нет вовсе
```

Сервер не поднимает HTTPS ни в каком виде "из коробки". Это значит в
открытом виде по сети идёт всё: логин/пароль при входе, JWT-токен (а с
ним — доступ ко всему на весь срок его жизни), содержимое загружаемых
документов и фото, и те самые пароли от хранилищ из п. 1.1. Для
мобильного клиента, который может работать через Wi-Fi в торговой
точке, это означает, что любой в той же сети может пассивно слушать
весь обмен.

Рекомендация: TLS-терминация хотя бы через reverse proxy (nginx +
Let's Encrypt/самоподписанный сертификат для LAN) перед сервером — уже
упомянуто как опциональный шаг в инструкции по установке, но сейчас это
не опция, а обязательный пункт для любого деплоя за пределами
полностью изолированной локальной сети.

## 2. Серьёзные находки

### 2.1 Нет защиты от brute-force на логине

20 быстрых попыток с неверным паролем подряд — все 20 получили `401` без
задержки, блокировки или капчи:
```
401 401 401 401 401 401 401 401 401 401 401 401 401 401 401 401 401 401 401 401
```
В сочетании с п. 1.2 (трафик не шифрован) и тем, что один скомпрометированный
admin-аккаунт открывает доступ вообще ко всему (включая пароли хранилищ
из п. 1.1) — это реальный путь компрометации всей системы по сети, не
только теоретический.

Рекомендация: `@nestjs/throttler` на `/api/auth/login` (например, 5
попыток в минуту на IP/username), в перспективе — блокировка аккаунта
после N неудачных попыток.

### 2.2 Две HIGH-уязвимости в зависимостях, которые реально едут в продакшен

`npm audit` по серверу — 26 уязвимостей всего, но большинство (ajv,
picomatch, tmp, webpack, inquirer, @angular-devkit/*) — это devDependencies
сборочного тулчейна (`@nestjs/cli` и его потроха), они не попадают в
`payload-windows` и не запускаются в реальном деплое. Но проверил
конкретно дерево *production*-зависимостей — там реально едут:

- **multer 2.0.2** (HIGH) — несколько DoS через переполнение/незавершённую
  очистку при загрузке файлов. Прямое попадание в самую используемую
  функцию приложения — `/api/files/upload`.
- **path-to-regexp 0.2.5/0.1.13** (HIGH) — ReDoS, тянется транзитивно
  через Express/`@nestjs/serve-static`, которым отдаётся и сама админка.
- `qs`/`uuid` (moderate) тоже в проде, но менее критичны.

Рекомендация: `npm audit fix --force` поднимет `@nestjs/platform-express`
и `@nestjs/serve-static` на мажорные версии — ломающее изменение,
требует отдельного прохода с регрессионным тестированием API, не
делал в рамках этого аудита (только проверка, не фикс).

### 2.3 Нет security-заголовков, фреймворк виден в открытую

```
HTTP/1.1 200 OK
X-Powered-By: Express
Access-Control-Allow-Origin: *
```
Нет `X-Frame-Options`, `X-Content-Type-Options`, `Content-Security-Policy`,
`Strict-Transport-Security` — ни одного. `X-Powered-By: Express` сам по
себе отдаёт атакующему информацию о стеке бесплатно. Поскольку этот же
процесс отдаёт и саму админ-панель (не только API), отсутствие
`X-Frame-Options`/CSP означает, что страницу логина админки в теории
можно встроить в `<iframe>` на чужом сайте (clickjacking).

Рекомендация: пакет `helmet`, один вызов `app.use(helmet())` в `main.ts`.

## 3. Средние находки

### 3.1 Сами документы/фото не шифруются при хранении

Ни локальный, ни FTP, ни Synology-адаптер не шифруют содержимое файла
перед записью — что было в буфере, то и легло на диск/NAS как есть. Для
системы, которая по смыслу спецификации (поле "место", документы и
фото с привязкой к точке) скорее всего имеет дело с реальными
документами клиентов, это означает: конфиденциальность данных целиком
зависит от того, насколько защищён сам диск/NAS — приложение со своей
стороны ничего не добавляет.

### 3.2 FTP-адаптер — только обычный FTP, без FTPS

`secure: false` зашито в коде адаптера без возможности включить
шифрование (в отличие от Synology-адаптера, где HTTPS уже добавлен).
Содержимое файлов и логин/пароль к самому FTP-серверу при записи через
этот адаптер идут открытым текстом до NAS/FTP-сервера отдельно от
проблемы 1.2 (это шифрование на "последней миле" до хранилища, а не
между клиентом и нашим сервером).

### 3.3 CORS открыт всем по умолчанию

```
Access-Control-Allow-Origin: *
```
получено в ответ на preflight с заголовком `Origin: https://evil-attacker-site.example`
— то есть сервер согласится на cross-origin запрос с ЛЮБОГО сайта, если
не задать `CORS_ORIGIN` в `.env` (опция уже есть с прошлого фикса, но не
обязательна и не включена по умолчанию). Поскольку токен лежит не в
cookie, а передаётся вручную в заголовке, классический CSRF тут не
работает напрямую — но открытый CORS всё равно увеличивает поверхность
атаки на будущее (например, при XSS где-то ещё) и просто плохая
гигиена для production.

### 3.4 Соединение с PostgreSQL — без TLS

В `typeorm.config.ts` нет `ssl`-опции вообще. Не критично, если БД на
той же машине (как в типовой инструкции по установке), но становится
проблемой в момент, когда `DB_HOST` указывает на БД на отдельном
сервере — трафик к ней тоже пойдёт в открытую.

## 4. Рекомендации (не дефекты, governance/процесс)

- Нет способа удалить загруженный файл/запись через UI — если когда-нибудь
  появятся клиенты из юрисдикции с правом на удаление персональных
  данных (GDPR и аналоги), сейчас удовлетворить такой запрос можно
  только руками через БД и хранилище напрямую.
- `synchronize: true` (вне `NODE_ENV=production`) — это про целостность
  данных, а не про конфиденциальность, но раз уж аудит "защиты данных" —
  стоит упомянуть: перед серьёзным продакшеном лучше перейти на
  миграции, синхронизация схемы на старте может неожиданно изменить
  структуру таблиц на реальных данных.
- Уязвимости esbuild/vite (admin-panel devDependencies) — не попадают в
  собранный `dist/`, реального риска для задеплоенного продукта нет;
  важно только если когда-нибудь `vite dev`-сервер окажется доступен
  по сети, а не только на машине разработчика.

## 5. Проверено и подтверждено безопасным

Не предположение — реально протестировано боевыми payload'ами:

- **SQL-инъекция**: `admin' OR '1'='1` и `admin'; DROP TABLE users; --`
  в логине, аналогичная инъекция в поле `place` при загрузке файла —
  везде корректный `401`/успешная безопасная обработка, таблицы целы,
  TypeORM-параметризация держит.
- **Подмена JWT**: смена `role` с `user` на `admin` в payload без
  пересборки подписи → `401`. Атака `alg=none` → `401`. Произвольный
  "мусорный" токен → `401`. Подпись реально проверяется, не только
  декодируется.
- **IDOR/эскалация прав**: обычный пользователь, пытающийся через прямой
  API-вызов сделать себя админом (`PATCH /api/users/1
  {"role":"admin"}`) — `403`, `RolesGuard` держит независимо от того,
  чей именно `id` указан в пути.
- **Утечка credentials через текст ошибки**: специально проверил — при
  падении FTP-записи (и при недоступном сервере, и при неверном
  пароле на доступном сервере) пароль хранилища НЕ попадает в текст
  ошибки, который видит клиент (`"530 Authentication failed."` /
  `"connect ECONNREFUSED..."` — без credentials). Более подробные
  сообщения об ошибках, добавленные в прошлом раунде фиксов, не создали
  новую утечку.
- `passwordHash` пользователей — не утекает ни через один проверенный
  эндпоинт (повторная регрессия после всех изменений).
- Path traversal через поле `place` — остаётся нейтрализован.

## 6. Верификация исправлений (17.06.2026)

Все пункты ниже исправлены и подтверждены реальным прогоном (не просто
код-ревью) на пересобранном сервере — том же payload, что зашит в
обновлённый `Setup.exe`.

**1.1 (credentials в открытом виде)** — поле `password` у storage
connections теперь шифруется AES-256-GCM перед записью в БД
(`common/crypto/encryption.util.ts`, ключ — `ENCRYPTION_KEY` в `.env`,
генерируется автоматически по той же схеме, что и `JWT_SECRET`).
Подтверждено прямым запросом к Postgres — в колонке лежит шифротекст,
не исходная строка. Поле полностью исключено из всех API-ответов
(create/list/update) — проверено: создание соединения с паролем не
возвращает пароль в ответе, список соединений тоже. Сквозной тест:
создание → правка БЕЗ повторной отправки пароля → реальная запись на
FTP всё равно прошла (расшифровка работает).

**1.2 (нет TLS)** — сервер теперь умеет поднимать HTTPS напрямую, без
reverse-proxy (`TLS_CERT_PATH`/`TLS_KEY_PATH` в `.env`). Проверено
настоящим самоподписанным сертификатом: обычный `http://` после этого
не принимает соединение вообще, `https://` без `-k` у curl корректно
отваливается как недоверенный (это и значит, что проверка сертификата
настоящая, не заглушка), `https://` с `-k` — полный цикл логин + запрос
с токеном проходит на 200.

**2.1 (нет защиты от brute-force)** — `@nestjs/throttler`, отдельный
жёсткий лимит на `/auth/login` (5 попыток в минуту). Проверено: 10
быстрых попыток подряд — после 5-й начинает отвечать `429`, считает все
запросы, а не только неудачные.

**2.2 (уязвимости в зависимостях)** — точечные `overrides` в
`package.json` на реально уязвимые версии в *production*-дереве:
`multer` 2.0.2→2.2.0, `path-to-regexp` (только внутри
`@nestjs/serve-static`) 0.2.5→3.3.0, `lodash` 4.17.21→4.18.1, `uuid`
9.0.1→11.1.1, `file-type` 20.4.1→21.3.2. При первой попытке override
`path-to-regexp` зацепил и собственную, отдельную (и уже безопасную)
зависимость самого Express — `npm ls` в чистой production-сборке
показал `invalid`. Поправлено явным вторым override, фиксирующим
Express на его родной `0.1.13`; повторная проверка в чистой
production-сборке (`scripts/package-windows.sh`) — без единого
`invalid`. `npm audit`: 26 → 19 уязвимостей; все оставшиеся —
подтверждённо либо dev-only тулчейн (`@nestjs/cli` и его зависимости),
либо требуют мажорного обновления NestJS без явного некритичного пути
(`express`/`body-parser`/`qs`-цепочка — версия `qs`, которая закрыла бы
уязвимость, не входит в диапазон, который `body-parser` готов принять;
оставлено как документированный остаточный риск).

**2.3 (нет security-заголовков)** — `helmet`, с кастомной CSP (иначе
сломались бы шрифты Google Fonts и инлайн-стили админки).
`X-Powered-By` снят явно. Проверено `curl -D-`: полный набор заголовков
появился (CSP, HSTS, X-Frame-Options, X-Content-Type-Options и т.д.),
`X-Powered-By` отсутствует.

**3.1 (документы не шифруются)** — добровольное шифрование содержимого
файлов at rest (`extraConfig.encryptAtRest` на storage connection,
выключено по умолчанию — не ломает обычный доступ к файлам напрямую с
NAS, если он не нужен). Заодно добавлены `GET /files/:id/download`
(админ — любой файл, обычный пользователь — только свой, с
расшифровкой на лету) и `DELETE /files/:id` (только админ, сначала
реально удаляет файл из хранилища и только потом запись в БД — если
удаление с диска не удалось, запись не трогается, чтобы можно было
повторить).

**3.2 (FTP без шифрования)** — `secure` теперь настраивается и для FTP
(FTPS), не только для Synology. Чекбокс в админке.

**3.3 (открытый CORS)** — без `CORS_ORIGIN` в `.env` сервер больше не
отвечает `*` кому угодно — разрешены только локальные адреса Vite
dev-сервера. Проверено: preflight с поддельным Origin не получает
заголовок `Access-Control-Allow-Origin` вообще, preflight с
`localhost:5173` получает.

**3.4 (Postgres без TLS)** — опциональный `DB_SSL=true` в `.env`,
по умолчанию выключен (не ломает обычный локальный Postgres).

Дополнительно: полный регрессионный прогон после всех фиксов подтвердил,
что старый функционал (загрузка документов/фото, ролевой доступ,
шаблоны именования) работает ровно как раньше, ничего не сломано.
