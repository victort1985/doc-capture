## 1. Критические дефекты

### 1.1 Обычный пользователь не может увидеть свою историю загрузок
`GET /api/files` (вкладка "История" в мобильном приложении) защищён
ролью `admin` на уровне контроллера. Любой пользователь с ролью `user`
— то есть весь штат, для которого и делалось мобильное приложение —
получает `403 Forbidden` при открытии вкладки "История".

Подтверждено: вход под учётной записью с ролью `user`, запрос
`GET /api/files` → `403 {"message":"Forbidden resource"}`.

Дополнительно: даже если снять ограничение по роли, сам эндпоинт не
фильтрует по текущему пользователю — обычный пользователь увидел бы
файлы **всех** пользователей, а не только свои. Это нужно чинить
вместе: добавить фильтр "только мои записи" для роли `user` и отдельно
оставить admin-only обзор всех файлов для админки.

### 1.2 Шаблоны имён файлов из админки никогда не применяются
В `files.service.ts` имя файла собирается по **захардкоженному**
шаблону `{date}_{place}_{docType}_{counter}` напрямую в коде. Записи
из таблицы `file_templates` (то, что создаётся/редактируется на
странице "Naming templates") нигде не читаются и не участвуют в
обработке загрузки.

Подтверждено: создан шаблон с уникальным паттерном
`CUSTOM_PATTERN_MARKER_{uuid}`, после чего выполнена загрузка — итоговое
имя файла не содержит маркер, использован стандартный паттерн.

Раздел админки полностью рабочий по CRUD, но не имеет никакого эффекта
на реальный результат — это особенно опасно, потому что выглядит
настроенным и работающим.

### 1.3 Загрузка настоящего PDF как "документа" роняет запрос с 500
`processDocument()` безусловно передаёт входной буфer в `sharp()`,
которая умеет читать только растровые изображения. Мобильный клиент
при этом явно разрешает выбрать `.pdf` из файлового менеджера
(`file_service.dart`, `allowedExtensions: [..., 'pdf']`).

Подтверждено: загрузка минимального валидного PDF с `docType=document`
→ `500 Internal Server Error`. В серверном логе настоящая причина:
`Error: Input buffer contains unsupported image format` — но клиент
этого текста никогда не видит (см. 2.1).

## 2. Серьёзные дефекты

### 2.1 Содержательные ошибки превращаются в голый "Internal server error"
`FilesService` в нескольких местах бросает обычный `Error(...)`, а не
`HttpException` Nest'а. Глобальный `HttpExceptionFilter` не умеет
отличить такую ошибку от настоящего краша и всегда отдаёт клиенту
`500 {"message":"Internal server error"}`, теряя оригинальный текст.

Подтверждено дважды:
- Загрузка без настроенного хранилища у пользователя → в логе
  `Error: No document storage connection configured for this user...`,
  клиенту — голый 500.
- Сбой обработки PDF (см. 1.3) → в логе точная причина от `sharp`,
  клиенту — тот же голый 500.

Это касается и мобильного приложения: пользователь увидит общее
"Upload failed, please try again" без единого шанса понять, что
конкретно не так (не настроено хранилище? битый файл? сеть?).
Нужно: оборачивать такие случаи в `BadRequestException`/
`UnprocessableEntityException` с понятным сообщением.

### 2.2 Нет таймаута на сетевые операции FTP/WebDAV-адаптеров
При тестировании с медленным/однопоточным тестовым WebDAV-сервером
запрос на загрузку зависал дольше 30 секунд без ответа клиенту, при
этом файл всё же физически записывался на диск с задержкой. Ни в
`webdav`-клиенте (`createClient(...)`), ни в обработчике запроса
не задан таймаут — недоступный или просто медленный NAS способен
подвесить запрос пользователя мобильного приложения на неопределённое
время без внятного сообщения об ошибке.

## 3. Дефекты средней важности

### 3.1 Нет проверки типа/содержимого файла при загрузке
`FilesInterceptor` принимает любой файл без `fileFilter` — несоответствие
реального содержимого и `docType` обнаруживается только постфактум как
краш внутри `sharp` (см. 1.3), а не как чистая ошибка валидации.

### 3.2 В админке нельзя отредактировать storage connection
Страница "Storage connections" — только создание и удаление. Опечатку
в host/port/пароле можно исправить только пересозданием подключения, что
дополнительно требует заново пройти Storage routing для каждого
пользователя, который на него ссылался.

### 3.3 Windows-инсталлятор не проверяет, не запущен ли сервер
Повторная установка/обновление через `Setup.exe` не предупреждает, если
`Start Doc Capture Server` уже открыт — файлы, заблокированные работающим
процессом Node на Windows, могут не обновиться без явной ошибки.

### 3.4 Деинсталлятор безусловно удаляет `app\.env`
`RMDir /r "$INSTDIR\app"` при удалении сносит файл с настройками БД и
`JWT_SECRET` без резервной копии и предупреждения — переустановка
(например, при обновлении) тихо обнуляет конфигурацию и обрывает все
текущие сессии (старые JWT перестанут проходить проверку после смены
секрета).

## 4. Наблюдения и рекомендации (не дефекты)

- Токен в админке хранится в `sessionStorage` без глобального
  перехватчика 401 — при истёкшей сессии человек видит баннер ошибки на
  текущей странице вместо автоматического перехода на экран входа.
- CORS открыт полностью (`app.enableCors()` без ограничения origin) —
  приемлемо для bearer-token API, но стоит сузить для продакшена.
- `JWT_SECRET=change_me` — дефолт уже описан в документации по установке,
  но стоит вынести в чек-лист как обязательный шаг, а не опциональный.
- Кнопка "Камера" в мобильном приложении по-прежнему открывает системную
  галерею/выбор фото (`image_picker`), а не живой превью камеры —
  известный TODO, подтверждён после дизайн-правок.
- Зависимость `flutter_image_compress` объявлена в `pubspec.yaml`, но
  нигде не используется — лишний вес приложения без пользы.
- После успешной загрузки поле "Место" на главном экране не очищается
  (очищаются только выбранные файлы) — возможно, осознанно для серии
  загрузок в одно место, но стоит подтвердить ожидаемое поведение.

## 5. Подтверждённая рабочая функциональность

Прогнано вживую, не предположение:

- Авторизация: успешный/неуспешный логин, блокировка отключённого
  (`isActive=false`) пользователя, корректные 401/403 на всех проверенных
  маршрутах.
- Users CRUD: минимальная длина пароля, `409` на дублирующийся username,
  `400` на неизвестное поле в теле запроса (whitelist-валидация работает).
- `passwordHash` не утекает ни через `/api/users`, ни через `/api/files`
  (регрессия по более раннему фиксу — подтверждена повторно).
- Адаптеры хранения — **все три** реально протестированы на настоящих
  локальных FTP- и WebDAV-серверах (не только заглушки): файлы доезжают,
  имена и вложенные подкаталоги формируются верно.
- Сжатие фото надёжно укладывается в ≤1МБ даже из исходника 3000×2000 в
  максимальном качестве.
- Лимит размера файла (20МБ) отрабатывает корректно (`413`).
- Попытка path traversal через поле `place` (`../../../etc/passwd`)
  нейтрализуется существующей санитизацией — файл остался внутри
  настроенного `basePath`, выхода за его пределы не произошло.
- Пакетная загрузка нескольких файлов в одном запросе верно увеличивает
  счётчик `{counter}` (001, 002, 003…).
- Удаление storage connection, на которую ссылается Storage routing
  пользователя, корректно обнуляет ссылку (`SET NULL`), а не падает и не
  оставляет битую связь.
- Фильтры журнала файлов (`?type=`) возвращают точные результаты.
- Все вызовы API из админ-панели после редизайна сверены с реальными
  маршрутами сервера — расхождений/опечаток в путях не найдено.
- ARB-файлы локализации (he/en/ru) полностью синхронны по набору ключей;
  захардкоженных нелокализованных строк в экранах Flutter не найдено
  (кроме намеренно англоязычного копирайта).
- Сборка сервера и admin-panel (TypeScript) проходит чисто после
  дизайн-правок.

## 6. Что не получилось протестировать в этой среде — явные ограничения

- **Мобильный клиент (Flutter)**: нет Flutter SDK и нет сетевого доступа
  к pub.dev в этом окружении — `flutter analyze`/`flutter test`/реальный
  запуск приложения не выполнялись. Всё, что касается мобильного клиента
  в этом отчёте — статический разбор кода, не прогон.
- **Admin-panel UI**: нет headless-браузера (chromium/playwright) в этой
  среде — реального клика по интерфейсу и визуальной проверки не было.
  Покрыто косвенно: чистая сборка TypeScript + сверка каждого API-вызова
  с реально протестированными серверными маршрутами.
- **Windows-инсталлятор**: собран и проверен структурно (настоящий PE32,
  логика `.nsi`-скрипта вычитана построчно), но никогда не запускался на
  реальной Windows-машине — поведение установки/удаления, ярлыков и
  проверки Node.js во время `.onInit` не подтверждено живым прогоном.
- **HTTPS до Synology**: чекбокс "Secure" корректно долетает до
  `extraConfig.secure` на уровне API, но реального HTTPS WebDAV-сервера
  для проверки настоящего TLS-хендшейка в этой среде не было.

## 8. Верификация исправлений (повторный прогон 17.06.2026)

Все пункты ниже — реально протестированы заново на пересобранном
сервере (тот же payload, что зашит в обновлённый `Setup.exe`), а не
просто "код выглядит исправленным".

**1.1** — `staff1` (role=user) → `GET /api/files` теперь `200`, видит
только свои записи (`0`, пока ничего не загружал), `passwordHash` через
join не утекает. Эндпоинт переехал в `FilesController`, роль `user`
всегда жёстко привязана к своему `userId`, `admin` — без ограничений.

**1.2** — создан шаблон `CUSTOM_PATTERN_MARKER_{counter}`, после загрузки
итоговое имя — `CUSTOM_PATTERN_MARKER_001.pdf`. Шаблон реально читается
из БД (`TemplatesService.findApplicableTemplate`, приоритет: свой шаблон
точного типа → свой "both" → общий точного типа → общий "both" → дефолт).

**1.3** — загрузка настоящего PDF как `docType=document` → `201`,
PDF передан как есть (тот же размер, что у исходника). Детект по
сигнатуре `%PDF` в начале буфера, без попытки прогнать через `sharp`.

**2.1** — загрузка без настроенного хранилища → `400` с реальным текстом
(`"No document storage connection configured for this user. Ask an admin
to set it up under Storage routing."`), а не голый 500. Та же обёртка
накрывает и сбои самой обработки файла, и сбои записи в хранилище.

**2.2** — при искусственно медленном WebDAV-сервере запрос теперь падает
чисто через 15 секунд с понятным сообщением о таймауте вместо зависания
без ответа. Таймаут настраивается через `STORAGE_OPERATION_TIMEOUT_MS`.

**3.1** — текстовый файл, выданный за фото → `400`
(`"fake.txt" can't be used as a photo (expected an image, got
text/plain)."`), без падения в `sharp`.

**3.2** — в админке у Storage connections появилось редактирование.
Проверено по-настоящему: подключение переименовано без передачи пароля
→ загрузка через него всё равно прошла (старый пароль не потёрся).

**3.3 / 3.4** — добавлены в `installer.nsi`: проверка занятого порта 3000
перед установкой (предупреждение, не блокирует) и резервная копия
`app\.env` в `$INSTDIR\.env.backup` перед удалением при анинсталле.
Логика проверена построчным разбором скрипта и успешной компиляцией в
`Setup.exe`; реального прогона на Windows, как и раньше, не было — нет
такой машины в этой среде.

**Дополнительно, не было в исходном отчёте, нашлось по ходу фиксов:**
`JWT_SECRET` из `.env` ранее **не применялся вообще** — из-за порядка
загрузки модулей сервер всегда подписывал токены литералом `change_me`
независимо от содержимого `.env` (подтверждено: токен, выданный с
`JWT_SECRET=мой_секрет` в `.env`, успешно проверялся ключом `change_me`).
Это серьёзнее, чем сама рекомендация поменять дефолт — менять его не
имело смысла. Исправлено: `.env` теперь гарантированно грузится до
инициализации модулей; если `JWT_SECRET` не задан или равен `change_me`,
сервер сам генерирует случайный секрет и сохраняет его обратно в `.env`
(переживает перезапуск, не разлогинивает всех на каждый рестарт).
Заодно: отключение пользователя теперь обрывает доступ по его текущему
токену немедленно, а не только блокирует будущие входы (раньше токен
оставался рабочим до истечения, до 7 дней).

Также по рекомендациям из раздела 4: добавлен `CORS_ORIGIN` (env,
сужает CORS без правки кода), в админке сессия восстанавливается при
перезагрузке страницы и автоматически сбрасывает на экран входа при
401 (раньше — нет ни того, ни другого), в мобильном клиенте кнопка
"Камера" теперь открывает настоящий live-превью (`CameraController`)
вместо системной галереи, после успешной загрузки очищается и поле
"Место", лишняя зависимость `flutter_image_compress` удалена.
