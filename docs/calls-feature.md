# Doc Capture — модуль "Вызовы" (קריאת שירות)

Новый функционал поверх существующего проекта: регистрация обращений
клиентов компании, статус-флоу, заметки и файлы по каждому вызову,
real-time уведомления. Старый функционал (загрузка документов/фото)
никуда не делся — в мобильном клиенте он переехал во вкладку
**"Переучет"**, рядом с новой вкладкой **"Вызов"**.

## 1. Что реализовано

### Сервер (NestJS)
Новый модуль `src/modules/calls/`:
- `POST /api/calls` — создать вызов (поля строго по спеке: место,
  опционально гео-координаты, срочность, кто создал — имя/должность/
  телефон, описание, чекбокс "необычный ущерб"). Автор в БД — текущий
  авторизованный пользователь, не из тела запроса.
- `GET /api/calls` — список **всех** вызовов, видно любому
  авторизованному пользователю (это общий рабочий список для всей
  команды, а не личный список создателя — соответствует пункту спеки
  "когда кто-то из пользователей выбирает вызов").
- `GET /api/calls/:id` — детали + заметки + вложения.
- `POST /api/calls/:id/status` — смена статуса (`open`/`in_progress`/
  `closed`). Кто и когда менял — фиксируется и отдаётся в ответе.
- `POST /api/calls/:id/notes` — текст и/или фото (multipart, поле
  `photo`).
- `POST /api/calls/:id/attachments` — файл (multipart, поле `file`) —
  идёт через тот же конвейер, что обычная загрузка документов: всё, что
  не PDF, гоняется через `processDocument()` (обрезка под документ +
  упаковка в PDF).

### Папка вызова
При создании вызова сразу формируется путь
`calls/{id}_{дата}_{место}` — туда пишутся и заметки (`notes/`), и
вложения (`files/`). При закрытии вызова (`status=closed`) к имени
папки добавляется юзернейм закрывшего, и физическая папка
переименовывается на каждом хранилище, где у вызова есть файлы
(`StorageAdapter.rename()` — реализовано для local/FTP/WebDAV).
Переименование — best-effort: если на каком-то хранилище не получилось
(например, оно недоступно), статус всё равно меняется, а в ответе
приходит `folderRenameWarning` с подробностями.

### Real-time уведомления
`NotificationsGateway` (Socket.IO, путь `/ws/notifications`,
авторизация тем же JWT) рассылает всем подключённым клиентам события:
`call:created`, `call:status_changed`, `call:note_added`,
`call:attachment_added`. Проверено живым тестом (подключение клиента +
создание/смена статуса/заметка/вложение — все 4 события долетели).

**Важное ограничение**: это уведомления, пока приложение открыто и
подключено — НЕ настоящий push от ОС (закрытое приложение ничего не
покажет). Реальный push потребовал бы Firebase Cloud Messaging с
собственным Firebase-проектом заказчика (ключи/конфиг, которые я не
могу создать сам) — сознательно не стал имитировать это
полу-рабочей заглушкой.

### Мобильное приложение
- `lib/screens/root_screen.dart` — новая внешняя навигация, 2 вкладки:
  "Переучет" (весь старый функционал, внутри — своя навигация
  Загрузка/История/Настройки через TabBar) и "Вызов"/קריאת שירות.
- `lib/screens/calls/` — список вызовов, форма создания (все поля
  спеки, кнопка геолокации опциональна), экран деталей (смена статуса,
  кто менял, заметки текст/фото, вложения камера/файлы).
- При входящем событии — `SnackBar` с текстом + автообновление списка.

### Что НЕ сделано в этом раунде
- **Админ-панель**: спека просила вкладку именно "в клиенте программы"
  (мобильное приложение) — её и сделал. Веб-админка пока без отдельной
  страницы управления вызовами; админ полноценно работает с вызовами
  через то же мобильное приложение (он обычный пользователь системы).
  Если нужна и веб-страница — это отдельный, тоже не маленький кусок
  работы, могу сделать по запросу.
- **cloudflared не запускался реально** — см. раздел 5 в
  `installation-guide.md`: скачать бинарник в этой среде не вышло
  (домен `release-assets.githubusercontent.com` не в разрешённом
  списке). Скрипты и инструкция написаны по документации Cloudflare, но
  проверены только синтаксически, не живым запуском.
- Поле `email` у пользователя добавлено (для документированной
  привязки доступа по email в Cloudflare Access), но в самой
  админ-панели поле пока не выведено в форму — добавляется напрямую
  через API при необходимости.

## 2. Сборка

Ничего нового по шагам — `npm install` в `server/` подтянет
`@nestjs/websockets`/`socket.io`, во `mobile-client/` —
`flutter pub get` подтянет `socket_io_client`/`geolocator`. Для Android/
iOS не забыть разрешение на геолокацию (раздел 3.2/4.2
`installation-guide.md`, рядом с разрешением на камеру).
